## 1. Overview

VIL provides 28 connector crates covering storage, databases, message queues, protocols, and triggers. Every connector follows the VIL Way pattern with three semantic macro annotations:

| Macro | Applied to | Generates |
|-------|-----------|-----------|
| `#[connector_fault]` | Error enums | `Display`, `Error`, `error_code()`, `kind()`, `is_retryable()` |
| `#[connector_event]` | Event structs | `#[repr(C)]`, `Default` (zeroed), compile-time size guard ≤192B |
| `#[connector_state]` | State structs | `#[repr(C)]`, `Default` (zeroed) |

These macros come from `vil_connector_macros` — a lightweight proc-macro crate with **zero VIL runtime dependencies** (only syn + quote). Compiles in ~2 seconds.

### Why Not `#[vil_fault]`?

`#[vil_fault]` from `vil_macros` provides full IR validation and Control Lane integration, but requires `vil_sdk` + `vil_ir` + `vil_validate` — a 5-crate dependency chain. For connector crates that only need error semantics (not lane routing), `#[connector_fault]` is the lightweight alternative.

| | `#[vil_fault]` | `#[connector_fault]` |
|---|---|---|
| **Use in** | Core, server, SDK crates | Connector, trigger, plugin crates |
| **Dependencies** | vil_macros → vil_sdk → vil_ir → vil_validate | vil_connector_macros (syn + quote only) |
| **Generates** | FaultHandler trait, Control Lane integration, IR entry | Display, error_code(), kind(), is_retryable() |
| **Compile impact** | ~10s (full IR validation) | ~2s (no IR) |

## 2. Connector Faults

### 2.1 Defining a Fault

```rust
use vil_connector_macros::connector_fault;

#[connector_fault]
pub enum MongoFault {
    ConnectionFailed { uri_hash: u32, reason_code: u32 },
    QueryFailed { collection_hash: u32, reason_code: u32 },
    InsertFailed { collection_hash: u32, reason_code: u32 },
    Timeout { collection_hash: u32, elapsed_ms: u32 },
    DeserializeFailed { collection_hash: u32 },
}
```

### 2.2 What the Macro Generates

```rust
// Auto-generated by #[connector_fault]:

impl MongoFault {
    /// Numeric error code (1-indexed by variant order).
    pub fn error_code(&self) -> u32 {
        match self {
            Self::ConnectionFailed { .. } => 1,
            Self::QueryFailed { .. } => 2,
            Self::InsertFailed { .. } => 3,
            Self::Timeout { .. } => 4,
            Self::DeserializeFailed { .. } => 5,
        }
    }

    /// Variant name as static string.
    pub fn kind(&self) -> &'static str {
        match self {
            Self::ConnectionFailed { .. } => "ConnectionFailed",
            Self::QueryFailed { .. } => "QueryFailed",
            // ...
        }
    }

    /// Whether this fault is retryable.
    /// Auto-detected: "Timeout", "ConnectionFailed", "Unavailable" → true.
    pub fn is_retryable(&self) -> bool {
        match self {
            Self::ConnectionFailed { .. } => true,
            Self::Timeout { .. } => true,
            _ => false,
        }
    }
}

impl std::fmt::Display for MongoFault { /* "QueryFailed(code=2)" */ }
impl std::error::Error for MongoFault {}
```

### 2.3 Rules

- All fields must be primitive types: `u8`, `u16`, `u32`, `u64`, `i32`, `i64`, `f32`, `f64`
- **No `String`, `Vec`, `Box`** — faults must be allocation-free
- **No `thiserror`** — the macro replaces it
- String context stored as `u32` hash via `vil_log::dict::register_str()`

### 2.4 Using with VIL Log

Faults integrate with the semantic log system:

```rust
pub async fn query(&self, collection: &str, filter: Document) -> Result<Vec<T>, MongoFault> {
    let start = std::time::Instant::now();
    let result = self.inner_query(collection, filter).await;
    let elapsed = start.elapsed();

    // Auto-emit db_log regardless of success/failure
    db_log!(Info, DbPayload {
        db_hash: self.db_hash,
        table_hash: register_str(collection),
        duration_us: elapsed.as_micros() as u32,
        error_code: match &result {
            Ok(_) => 0,
            Err(f) => f.error_code() as u8,  // ← from #[connector_fault]
        },
        ..Default::default()
    });

    result
}
```

## 3. Connector Events

Events represent **what happened** in a connector — used for Tri-Lane Data Lane emission and pipeline reactivity.

### 3.1 Defining an Event

```rust
use vil_connector_macros::connector_event;

#[connector_event]
pub struct MongoDocumentInserted {
    pub collection_hash: u32,
    pub document_id_hash: u32,
    pub size_bytes: u32,
    pub timestamp_ns: u64,
}

#[connector_event]
pub struct S3ObjectCreated {
    pub bucket_hash: u32,
    pub key_hash: u32,
    pub size_bytes: u64,
    pub etag_hash: u32,
    pub timestamp_ns: u64,
}
```

### 3.2 What the Macro Generates

```rust
#[derive(Debug, Clone, Copy)]
#[repr(C)]
pub struct MongoDocumentInserted { /* fields */ }

impl Default for MongoDocumentInserted {
    fn default() -> Self { unsafe { std::mem::zeroed() } }
}

// Compile-time size guard
const _: () = {
    assert!(
        std::mem::size_of::<MongoDocumentInserted>() <= 192,
        "connector event must fit within 192 bytes (LogSlot payload)"
    );
};
```

### 3.3 Rules

- Must be `≤192 bytes` (fits in LogSlot payload) — compile error if exceeded
- `#[repr(C)]` for deterministic memory layout
- All fields must be primitive or fixed-size arrays
- `u32` hashes for string-derived fields

## 4. Connector State

State represents **live health and metrics** for a connector — used by ServiceProcess for monitoring.

### 4.1 Defining State

```rust
use vil_connector_macros::connector_state;

#[connector_state]
pub struct MongoPoolState {
    pub active_connections: u32,
    pub idle_connections: u32,
    pub waiting_requests: u32,
    pub total_queries: u64,
    pub total_errors: u64,
    pub avg_latency_us: u32,
}
```

### 4.2 Integration with ServiceProcess

```rust
use std::sync::Arc;
use vil_db_mongo::{MongoClient, MongoConfig, MongoPoolState};

// In VilApp setup:
let mongo = vil_db_mongo::process::create_client(config).await?;

VilApp::new("my-service")
    .service(ServiceProcess::new("mongo")
        .state(mongo.clone())
        .state(Arc::new(MongoPoolState::default()))  // live metrics
        .endpoint(Method::POST, "/api/users", post(create_user)))
    .run().await;
```

## 6. Complete Example: MongoDB with Full Semantics

```rust
use std::sync::Arc;
use vil_server::prelude::*;
use vil_db_mongo::{MongoClient, MongoConfig, MongoFault, MongoPoolState};
use vil_db_mongo::events::MongoDocumentInserted;

// Handler using full VIL Way + connector semantics
async fn create_user(
    ctx: ServiceCtx,
    body: ShmSlice,
) -> Result<VilResponse<User>, MongoFault> {
    let mongo = ctx.state::<Arc<MongoClient>>()?;
    let input: CreateUser = ctx.parse(&body)?;

    // Insert — auto-emits db_log! with timing
    let id = mongo.insert_one("users", &input).await?;

    let user = User { id, name: input.name, email: input.email };

    // Event would flow on Data Lane in a full Tri-Lane setup
    // ctx.emit(&MongoDocumentInserted {
    //     collection_hash: register_str("users"),
    //     document_id_hash: register_str(&id),
    //     size_bytes: std::mem::size_of::<User>() as u32,
    //     timestamp_ns: now_ns(),
    // }).await;

    Ok(VilResponse::created(user))
}

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    // Init VIL log
    vil_log::runtime::init_logging(
        vil_log::LogConfig {
            ring_slots: 1 << 20,
            level: vil_log::LogLevel::Info,
            batch_size: 1024,
            flush_interval_ms: 100,
            threads: Some(4),
        },
        vil_log::StdoutDrain::resolved(),
    );

    // Create MongoDB client
    let mongo = vil_db_mongo::process::create_client(MongoConfig {
        uri: "mongodb://localhost:27017".into(),
        database: "myapp".into(),
        ..Default::default()
    }).await?;

    // Run VilApp
    VilApp::new("user-service")
        .port(3080)
        .service(ServiceProcess::new("users")
            .state(mongo)
            .state(Arc::new(MongoPoolState::default()))
            .endpoint(Method::POST, "/api/users", post(create_user)))
        .run().await;

    Ok(())
}
```
