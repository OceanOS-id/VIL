# VIL Custom Code Guide — Native, WASM, and Sidecar

VIL supports 3 execution modes for custom business logic. Each mode has different trade-offs for performance, isolation, and developer experience.

## Execution Modes

| Mode | ExecClass | Language | Overhead | Isolation | Hot-Deploy |
|------|-----------|----------|----------|-----------|------------|
| **Native** | `Native` | Rust | 0 (compiled) | None (full process) | No (recompile) |
| **WASM** | `WasmFaaS` | Rust → .wasm, AssemblyScript | ~1-5μs | Memory sandbox | Yes |
| **Sidecar** | `SidecarProcess` | Python, Go, Java, TypeScript, any | ~12μs | Full process | Yes |

### Decision Guide

| Scenario | Recommended Mode |
|----------|-----------------|
| Maximum performance, no isolation needed | **Native** |
| Business rules that change frequently | **WASM** |
| Existing Python/Go/Java codebase | **Sidecar** |
| Security boundary (untrusted code) | **WASM** |
| ML inference with Python libraries | **Sidecar** |
| Same logic, need comparison benchmark | All 3 (see example below) |

## YAML Manifest

### Native Code (Rust handler)

```yaml
nodes:
  enricher:
    type: process
    exec_class: AsyncTask
    code:
      mode: handler
      handler: enrich_credit_data
      async: true
      state_access: readwrite
```

Generated Rust:
```rust
async fn enrich_credit_data(ctx: ServiceCtx, body: ShmSlice) -> VilResponse<EnrichedRecord> {
    // Your native Rust logic here
}
```

### WASM Module

```yaml
vil_wasm:
  - name: pricing
    language: rust                          # or "assemblyscript"
    wasm_path: ./wasm-modules/pricing.wasm  # pre-compiled .wasm binary
    pool_size: 4                            # pre-warmed instance pool
    functions:
      - name: calculate_price
        input: { base: i32, quantity: i32 }
        output: i32
        description: "Calculate price with volume discount"
      - name: validate_order
        input: { amount: i32, limit: i32 }
        output: i32
        description: "Returns 1 if valid, 0 if invalid"
    sandbox:
      timeout_ms: 5000
      max_memory_mb: 64

# Reference in a node:
nodes:
  price_check:
    type: process
    code:
      mode: wasm
      module: pricing                       # references vil_wasm[].name
      function: calculate_price
```

### Sidecar (External Process)

```yaml
sidecars:
  - name: ml-scorer
    command: python3
    script: ./sidecars/ml_scorer.py         # relative to project root
    protocol: uds                           # Unix Domain Socket
    socket: /tmp/vil-ml-scorer.sock         # optional, auto-generated
    methods:
      - predict
      - score_batch
    auto_restart: true
    max_concurrent: 10
    startup_timeout_ms: 5000

  - name: fraud-detector
    command: go run
    script: ./sidecars/fraud_detector.go
    methods:
      - check_fraud
      - risk_score

# Reference in a node:
nodes:
  scoring:
    type: process
    exec_class: SidecarProcess
    code:
      mode: script
      runtime: sidecar
      source: ml-scorer                     # references sidecars[].name
```

## Complete Example: All 3 Modes

```yaml
vil_version: "0.6.0"
name: hybrid-pipeline
port: 8080

# ── WASM modules ──
vil_wasm:
  - name: validation
    wasm_path: ./wasm-modules/validation.wasm
    pool_size: 4
    functions:
      - name: validate_record
        description: "Validate credit record fields"

# ── Sidecar processes ──
sidecars:
  - name: ml-scorer
    command: python3
    script: ./sidecars/ml_scorer.py
    methods: [predict]
    auto_restart: true

# ── Endpoints ──
endpoints:
  # Native Rust handler — maximum performance
  - method: POST
    path: /api/enrich
    handler: enrich_handler
    exec_class: AsyncTask
    input:
      type: CreditRecord
    output:
      type: EnrichedRecord

  # WASM handler — sandboxed validation
  - method: POST
    path: /api/validate
    handler: validate_handler
    exec_class: WasmFaaS

  # Sidecar handler — Python ML inference
  - method: POST
    path: /api/score
    handler: score_handler
    exec_class: SidecarProcess
```

### Project Structure

```
my-project/
├── vil.yaml                    # Pipeline/server manifest
├── wasm-modules/
│   ├── build-wasm.sh           # Compile Rust → .wasm
│   ├── validation/
│   │   ├── Cargo.toml
│   │   └── src/lib.rs          # #[no_mangle] pub extern "C" fn validate_record(...)
│   └── out/
│       └── validation.wasm     # Compiled WASM binary
├── sidecars/
│   ├── ml_scorer.py            # Python sidecar (uses vil_sidecar SDK)
│   └── fraud_detector.go       # Go sidecar
└── src/
    └── main.rs                 # Generated by `vil compile` or hand-written
```

## Compile & Run

```bash
# Compile YAML to native binary
vil compile --input vil.yaml --output my-app --release

# Or run directly
vil run vil.yaml

# The binary:
# - Loads .wasm files from wasm-modules/ at startup
# - Spawns sidecar processes (Python/Go) automatically
# - Native handlers are compiled in
./my-app
```

## For Non-Rust Developers

| Language | Pipeline Definition | Custom Business Logic |
|----------|--------------------|-----------------------|
| **Python** | Transpile SDK → YAML → binary | **Sidecar** (UDS) |
| **Go** | Transpile SDK → YAML → binary | **Sidecar** (UDS) |
| **Java** | Transpile SDK → YAML → binary | **Sidecar** (UDS) |
| **TypeScript** | Transpile SDK → YAML → binary | **WASM** (AssemblyScript) or **Sidecar** |

The transpile SDK generates **pipeline routing/topology only** — not business logic. Custom code executes via WASM or Sidecar.

### Python Sidecar Example

```python
# sidecars/ml_scorer.py
from vil_sidecar import VilSidecar

sidecar = VilSidecar("ml-scorer")

@sidecar.method("predict")
def predict(data: dict) -> dict:
    score = my_ml_model.predict(data["features"])
    return {"score": score, "confidence": 0.95}

sidecar.run()
```

### Go Sidecar Example

```go
// sidecars/fraud_detector.go
package main

import vil "github.com/oceanos-id/vil-sidecar-go"

func main() {
    s := vil.NewSidecar("fraud-detector")
    s.Method("check_fraud", func(input []byte) ([]byte, error) {
        // Your Go business logic
        return []byte(`{"fraud": false}`), nil
    })
    s.Run()
}
```

## Failover Chain

WASM and Sidecar can be combined with failover:

```yaml
failover:
  entries:
    - primary: native-handler
      backup: wasm-fallback
      strategy: instant
    - primary: wasm-fallback
      backup: sidecar-fallback
      strategy: instant
      condition: WasmExecutionFailed
```

Failover path: **Native → WASM → Sidecar** — degradation in performance but not in availability.

## Performance Comparison

Measured on Intel i9-11900F (8C/16T), same business logic (credit validation):

| Mode | req/s | P50 | P99 | Notes |
|------|-------|-----|-----|-------|
| Native | ~40,000 | 0.5ms | 25ms | Compiled in, zero overhead |
| WASM | ~35,000 | 0.6ms | 28ms | Pre-warmed pool, 1 memory copy |
| Sidecar | ~8,000 | 2ms | 15ms | UDS IPC, descriptor serialization |

Native and WASM are near-identical for compute-bound logic. Sidecar overhead is dominated by IPC serialization — acceptable for I/O-bound tasks (ML inference, external API calls).
